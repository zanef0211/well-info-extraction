## 一、设计目标

### 1.1 核心目标

本提示词系统的核心目标是为大语言模型（LLM）提供清晰、准确、专业的指令，使其能够从油气井技术文档中精确提取结构化信息。

**具体目标：**

1. **准确性**: 准确识别并提取文档中的关键字段
2. **专业性**: 保持油气井领域专业术语的一致性和准确性
3. **结构化**: 输出符合预定义JSON Schema的标准化格式
4. **置信度评估**: 对提取结果的可靠性进行量化评估
5. **容错性**: 能够处理文档不完整、模糊或冲突的情况

### 1.2 应用场景

提示词系统覆盖以下核心应用场景：

- **文档分类**: 自动识别文档类型（钻井、录井、测井等）
- **井号识别**: 从文档中提取和规范化井号
- **字段提取**: 按分类提取专业字段
  - 基本信息 (basic)
  - 钻井资料 (drilling)
  - 录井资料 (mudlogging)
  - 测井资料 (wireline)
  - 试油测试资料 (testing)
  - 完井资料 (completion)
  - 地质资料 (geology)
  - 生产资料 (production)

## 二、设计原则

### 2.1 通用设计原则

#### 原则1: 清晰性 (Clarity)

**定义**: 提示词必须清晰、简洁、无歧义

**实现方式**:
- 使用明确的指令动词："提取"、"识别"、"判断"
- 避免模糊表达："可能"、"大概"、"似乎"
- 明确输出格式要求："请只输出JSON"
- 提供明确的字段定义和示例

**示例**:
```python
# ✅ 好的提示词
"从以下文档中提取井号、井深、开钻日期。严格按照指定的JSON格式输出。"

# ❌ 差的提示词
"看看文档里有没有井号、井深这些信息，如果有就提取出来。"
```

#### 原则2: 上下文感知 (Context Awareness)

**定义**: 利用文档内容和元数据提供充分的上下文信息

**实现方式**:
- 在提示词中提供完整的文档内容预览
- 明确标注文档类型和分类
- 提供字段定义和取值范围
- 说明字段之间的关联关系

**示例**:
```python
# ✅ 好的提示词
"""
从以下钻井日报文档中提取关键字段。

文档内容预览:
{document_text}

提取字段:
- RigModel (钻机型号): 钻井使用的钻机型号
- TotalDepth (完钻井深,单位: 米): 完钻时的实际井深
...
"""

# ❌ 差的提示词
"""
提取下面的字段:
- 钻机型号
- 井深
"""
```

#### 原则3: 结构化输出 (Structured Output)

**定义**: 强制模型输出符合预定义Schema的结构化数据

**实现方式**:
- 明确指定JSON格式要求
- 提供完整的JSON Schema示例
- 要求字段包含value和confidence
- 强制只输出JSON，禁止其他内容

**示例**:
```python
# 输出要求
"""
输出JSON格式:
{
  "WellNo": {"value": "井号", "confidence": 0.95},
  "RigModel": {"value": "钻机型号", "confidence": 0.90}
}

请只输出JSON,不要有其他内容。
"""
```

#### 原则4: 置信度评估 (Confidence Assessment)

**定义**: 对提取结果的可靠性进行量化评估

**置信度分级**:
```
0.95-1.0: 文档中明确提到，非常确定
0.80-0.95: 文档中提到，较为确定
0.60-0.80: 需要推断，有一定不确定性
0.30-0.60: 不确定，需要人工审核
<0.30: 可能错误
```

**实现方式**:
- 根据字段的明确程度和出现次数设定置信度
- 对于数值字段，如果文档中直接给出，置信度设为0.95+
- 对于推断字段，置信度设为0.60-0.80
- 对于不确定字段，置信度设为<0.30

#### 原则5: 领域专业性 (Domain Specificity)

**定义**: 保持油气井领域的专业术语准确性

**实现方式**:
- 使用标准化的油气井专业术语
- 提供字段的详细定义和单位
- 包含典型值的范围和约束
- 说明字段的数据类型要求

**示例**:
```
字段: TotalDepth
- 正确表达: 完钻井深 (单位: 米)
- 错误表达: 深度、井深

字段: OilRate
- 正确表达: 日产油量 (单位: m³/d)
- 错误表达: 产油量、油量
```

### 2.2 分类提示词设计原则

#### 原则1: 分类清晰化

**定义**: 明确区分一级分类和二级文档分类

**一级分类** (7大类):
- `drilling`: 钻井资料
- `mudlogging`: 录井资料
- `wireline`: 测井资料
- `testing`: 试油-测试资料
- `completion`: 完井资料
- `geology`: 地质资料
- `production`: 生产资料

**二级文档分类** (每个分类包含多种文档类型):
- 钻井资料: 钻井设计、完井设计、钻井日报、固井报告等
- 录井资料: 录井报告、气测记录、岩屑录井等

**实现方式**:
```python
"""
可选分类(一级分类):
- drilling: 钻井资料 (包含钻头程序、套管程序、固井数据等)
- mudlogging: 录井资料 (包含气测数据、岩屑描述、钻时数据等)
...

二级文档分类(请同时识别):
- 钻井资料: 钻井设计/完井设计/钻井日报/固井报告/钻井工程总结/井身结构图...
- 录井资料: 录井报告/气测记录/岩屑录井/综合录井图/钻时录井...
"""
```

#### 原则2: 证据链推理

**定义**: 要求模型在输出分类时提供推理依据

**实现方式**:
```python
"""
输出JSON格式:
{
  "category": "一级分类代码",
  "doc_category": "二级文档分类",
  "confidence": 0.95,
  "evidence": "分类依据简述"
}
"""
```

### 2.3 字段提取提示词设计原则

#### 原则1: 字段完整性

**定义**: 为每个分类提供全面的字段定义

**实现方式**:
```python
"""
提取字段:
- WellNo (井号)
- DocCategory (文档分类: basic)
- Oilfield (油田/区块)
- WellType (井别)
- SpudDate (开钻日期,格式: YYYY-MM-DD)
- TotalDepth (完钻井深,单位: 米)
...
"""
```

#### 原则2: 单位和格式规范

**定义**: 明确指定数值字段的单位和日期格式

**实现方式**:
```python
"""
- SpudDate (开钻日期,格式: YYYY-MM-DD)
- TotalDepth (完钻井深,单位: 米)
- OilRate (日产油量,单位: m³/d)
- WaterCut (含水率,单位: %)
"""
```

#### 原则3: 字段间关联

**定义**: 说明字段之间的关联关系和约束条件

**示例**:
```
关联字段:
- SpudDate < CompletionDate < WellCompletionDate (时间顺序)
- DesignDepth >= TotalDepth (设计深度应大于等于完钻深度)
- DrillingDays = CompletionDate - SpudDate (钻井周期可计算)
```

### 2.4 错误处理原则

#### 原则1: 缺失处理

**定义**: 当文档中缺少某些字段时，不强制输出

**实现方式**:
- 只提取文档中明确存在的字段
- 缺失的字段不在JSON中出现
- 不生成虚构值

#### 原则2: 矛盾处理

**定义**: 当文档中同一字段有多个冲突值时，选择最合理的

**实现方式**:
- 优先选择文档中明确的定义值
- 查看出现频率，出现多次的可能更准确
- 设置较低的置信度表示不确定性

#### 原则3: 格式错误处理

**定义**: 对于格式不规范的值，尝试修正或标记

**实现方式**:
- 日期格式不统一时，尝试识别并标准化为 YYYY-MM-DD
- 数值单位缺失时，根据上下文推断
- 如果无法推断，标记置信度为0.30-0.60

## 三、提示词架构

### 3.1 提示词层级

```
prompts.py
├── SYSTEM_PROMPT              # 全局系统提示词
├── CLASSIFY_PROMPT           # 文档分类提示词
├── WELLNO_EXTRACT_PROMPT    # 井号识别提示词
├── BASIC_EXTRACT_PROMPT      # 基本信息提取提示词
├── DRILLING_EXTRACT_PROMPT  # 钻井资料提取提示词
├── MUDLOGGING_EXTRACT_PROMPT # 录井资料提取提示词
├── WIRELINE_EXTRACT_PROMPT   # 测井资料提取提示词
├── TESTING_EXTRACT_PROMPT    # 试油测试资料提取提示词
├── COMPLETION_EXTRACT_PROMPT # 完井资料提取提示词
├── GEOLOGY_EXTRACT_PROMPT    # 地质资料提取提示词
├── PRODUCTION_EXTRACT_PROMPT # 生产资料提取提示词
└── PROMPT_MAP              # 提示词映射字典
```

### 3.2 提示词映射机制

```python
PROMPT_MAP: Dict[str, str] = {
    "basic": BASIC_EXTRACT_PROMPT,
    "drilling": DRILLING_EXTRACT_PROMPT,
    "mudlogging": MUDLOGGING_EXTRACT_PROMPT,
    "wireline": WIRELINE_EXTRACT_PROMPT,
    "testing": TESTING_EXTRACT_PROMPT,
    "completion": COMPLETION_EXTRACT_PROMPT,
    "geology": GEOLOGY_EXTRACT_PROMPT,
    "production": PRODUCTION_EXTRACT_PROMPT,
}
```

**使用方式**:
```python
from config.prompts import Prompts

# 根据分类获取对应的提示词
category = "drilling"
prompt = Prompts.PROMPT_MAP[category]
```

### 3.3 提示词调用流程

```
文档输入
    ↓
文档分类 (CLASSIFY_PROMPT)
    ↓
确定一级分类 (category) 和二级分类 (doc_category)
    ↓
井号识别 (WELLNO_EXTRACT_PROMPT)
    ↓
提取井号并归一化 (normalized_well)
    ↓
字段提取 (PROMPT_MAP[category])
    ↓
结构化JSON输出
```

## 四、字段库集成

### 4.1 与字段库的关系

提示词与字段库(`config/field_schemas.py`)紧密关联：

- 字段名称必须与字段库定义一致
- 字段类型、单位、取值范围必须匹配字段库
- 置信度评估参考字段库的必填/可选属性

### 4.2 字段验证机制

基于提示词提取的结果，通过字段库进行双向验证：

1. **名称验证**: 提取的字段名必须在字段库中
2. **类型验证**: 字段值类型必须符合字段库定义
3. **范围验证**: 数值必须在字段库定义的[min_value, max_value]范围内
4. **必填验证**: 必填字段的缺失应降低置信度

## 五、优化建议

### 5.1 持续优化方向

1. **Few-shot Learning**: 在提示词中添加示例
   ```python
   """
   示例:
   文档: "井号: TS1-1, 完钻井深: 5230米"
   输出: {"WellNo": {"value": "TS1-1", "confidence": 0.95}, ...}
   """
   ```

2. **思维链 (Chain-of-Thought)**: 引导模型逐步思考
   ```python
   """
   分析步骤:
   1. 首先识别文档类型
   2. 然后定位关键字段
   3. 接着提取具体值
   4. 最后评估置信度
   """
   ```

3. **错误示例**: 在提示词中提供错误示例避免
   ```python
   """
   错误示例:
   - 不要编造文档中不存在的字段
   - 不要忽略明确提到的单位
   - 不要混淆"设计井深"和"完钻井深"
   """
   ```

### 5.2 性能监控指标

- **提取准确率**: 目标 >90%
- **字段覆盖率**: 目标 >85%
- **置信度校准**: 置信度与实际准确性偏差 <10%
- **响应时间**: 单文档提取 <30秒

### 5.3 版本管理

每次修改提示词后：
1. 更新文档版本号
2. 在变更说明中记录修改内容
3. 通过A/B测试验证效果
4. 保留历史版本以支持回滚

## 七、维护指南

### 7.1 修改提示词的流程

1. **识别需求**: 确定需要修改的提示词部分
2. **分析影响**: 评估修改对现有功能的兼容性
3. **编写提示词**: 遵循设计原则编写新的提示词
4. **测试验证**: 使用真实文档测试提取效果
5. **更新文档**: 记录修改原因和效果
6. **版本发布**: 更新版本号并通知相关开发

### 7.2 常见问题处理

**问题1: 提取准确率下降**
- 检查提示词是否清晰
- 增加字段定义的详细程度
- 添加Few-shot示例

**问题2: 置信度评估不准确**
- 调整置信度分级标准
- 增加对文档明确程度的判断说明
- 校准置信度与实际准确性的对应关系

**问题3: 特定字段漏检**
- 检查字段定义是否完整
- 添加同义词提示
- 增加上下文关联说明
