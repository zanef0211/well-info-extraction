## 项目概述

### 项目定位

本项目是一个基于AI的油气井关键信息提取系统,旨在将非结构化的油气井资料(钻井、录井、测井、试油/测试、完井、地质、生产等)自动转换为结构化数据,支持一键导入业务系统。

### 核心价值

- **效率提升**: 单井资料整理时间从 4-8小时降至 <30分钟
- **数据质量**: 数据入库齐全率从 ~60% 提升至 ≥90%
- **成本节约**: 人力成本节省 ≥70%
- **智能化**: AI驱动的自动分类、识别、提取、校验

### 技术特色

- **多模态处理**: 支持 PDF、Word、Excel、PPT、TXT 等多种格式
- **智能分类**: 8大类文档自动分类,准确率 ≥90%
- **精准抽取**: 167个核心字段结构化提取,关键字段准确率 ≥90%
- **质量保障**: 多维度校验机制,置信度评估,人工审核闭环
- **灵活部署**: 支持API调用和私有化部署两种模式

## 核心业务场景

### 场景1: 单口井批量资料一键入库

**用户流程**: 批量上传该井全部资料 → AI自动分类 → 按标准字段提取关键数据 → 校验清洗 → 人工审核 → 一键写入业务系统/数据中台

**价值**: 实现一次上传、全量利用

### 场景2: 多口井混合资料批量处理

**用户流程**: 批量上传多井混合资料 → AI自动识别井号 → 按井归组 → 逐井执行分类→提取→校验 → 批量入库 → 生成多井汇总报告

**价值**: 解决"多井混传、人工分拣"痛点

### 场景3: 历史资料治理与补录

**用户流程**: 批量导入历史纸质扫描件/电子老文档 → AI结构化提取 → 补齐缺失字段 → 修正冲突数据 → 标准化入库

**价值**: 盘活存量资产,支撑老井潜力再评价、储量复算、方案优化

## 技术栈选型

### 整体技术路线

```
文档输入
   ↓
【Stage 1】文档预处理 (OCR + 版面解析)
   ↓
【Stage 2】文档分类模型 (8分类)
   ↓
【Stage 3】井号识别模型 (NER)
   ↓
【Stage 4】关键字段抽取模型 (结构化信息抽取)
   ↓
【Stage 5】校验与置信度评估 (规则 + 打分)
   ↓
结构化 JSON 输出 (送人工审核 / 入库)
```

### 核心技术组件

#### 1. 文档预处理
- **OCR引擎**: PaddleOCR v4 (中文支持优秀,专业版对手写字体友好)
- **PDF解析**: PyMuPDF (文字型/扫描型PDF均支持,表格提取)
- **文档解析**: python-docx (Word)、openpyxl (Excel)、python-pptx (PPT)

#### 2. 文档分类
- **Embedding模型**: BGE-M3 (零样本效果好,快速上线)
- **Fine-tune模型**: BERT-wwm-ext (有标注数据≥200条后提升至95%+)
- **兜底机制**: 关键词匹配算法

#### 3. 井号识别
- **优先策略**: 正则规则 (覆盖主流井号格式,80%+覆盖)
- **兜底策略**: LLM (ChatGLM3-6B / Qwen2.5-7B)
- **井号格式**: 
  - 中文名称: 塔中1井、苏里格6-3-5井
  - 字母数字: TS1-1、HZ21-3D、QHD32-6-1
  - 区块+序号: GS17-1-4、PL19-3-1

#### 4. 字段抽取
- **API模式**: Qwen2.5-72B / DeepSeek-V3 (成本 ~0.002元/千token,单文档 ~0.05元)
- **私有化部署**: Qwen2.5-14B-Instruct (A100/H20 × 2,延迟 ~3s/文档,吞吐 ~20文档/分钟)

#### 5. 规则校验
- **数值范围**: 产量、压力、深度等数值合理性检查
- **日期格式**: 支持 YYYY-MM-DD 等多种格式
- **必填字段**: WellNo、DocCategory 必须非空
- **井号格式**: 长度2-30字符检查

#### 6. 后端服务
- **Web框架**: FastAPI (高性能异步框架)
- **并发处理**: ThreadPoolExecutor (支持批量并发处理)

**接口路径**: `POST /extract/single`

**功能**: 上传单个文档,返回结构化抽取结果

**输入**: 文件 (PDF/Word/Excel等)

**输出**: JSON
```json
{
  "status": "success",
  "well_no": "TS1-1",
  "doc_category": "drilling",
  "overall_confidence": 0.92,
  "extracted_fields": {
    "WellNo": {"value": "TS1-1", "confidence": 0.95},
    "DocCategory": {"value": "drilling", "confidence": 1.0},
    "RigModel": {"value": "ZJ50DB", "confidence": 0.85},
    ...
  },

  "processing_time_ms": 12500

```

## 关键决策点

### 1. 部署方式选择

**问题**: 是否需要私有化部署?

**决策依据**:
- **数据敏感度高** → 私有化部署 Qwen2.5-14B-Instruct (A100/H20 × 2)
- **数据敏感度低** → 调用 DeepSeek-V3 / Qwen-Max API

**对比**:
| 对比项 | 私有化部署 | API 调用 |
|-------|-----------|---------|
| 成本 | 硬件成本高(2张A100 ~10万),运行成本约0.002元/千token | 无硬件成本,运行成本约0.05元/文档 |
| 数据安全 | 完全安全 | 数据需上传到云端 |
| 延迟 | ~3s/文档 | ~2-5s/文档(取决于网络) |
| 吞吐量 | ~20文档/分钟 | 受限于API速率限制 |
| 维护成本 | 需要运维团队 | 无需运维 |

### 2. 模型迭代策略

**阶段0 (上线初期)**: 纯提示词工程
- 文档分类准确率: ~85%
- 字段抽取准确率: ~75%
- 人工审核率: ~40%

**阶段1 (2周后,有标注数据)**: 提示词优化 + 少样本示例
- 分类: ~90%,字段抽取: ~82%,审核率降至 25%

**阶段2 (6周后,≥500条标注)**: Fine-tune 分类模型
- 分类: ~96%,字段抽取: ~88%,审核率降至 15%

**阶段3 (持续)**: 专业领域 SFT + RLHF
- 字段抽取: ~93%+,审核率降至 <10%

### 3. 井号识别策略

**优先策略**: 正则规则
- 覆盖主流井号格式: 中文+数字、字母+数字、区块+序号
- 覆盖率: 80%+
- 速度: <0.1秒

**兜底策略**: LLM
- 处理异常格式
- 归一化井号
- 成本: ~0.01元/文档

## 风险与应对

### 技术风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| OCR 识别准确率不足（手写字体、模糊扫描件） | 低置信度字段多,人工审核量大 | 推荐使用 PaddleOCR 专业版,支持图像预处理 |
| LLM 抽取结果不稳定 | 数据不可信 | 设置低 temperature(0.05),使用确定性输出 |

### 数据风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 样本数据质量差（标注错误、样本覆盖不全） | 模型训练效果差 | 建立数据质量审核流程,定期清洗样本 |

### 业务风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 业务规则变化（字段定义、标准更新） | 系统失效 | 设计灵活的字段库管理模块,支持动态更新 |

### 安全风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 数据泄露（文档上传到云端 API） | 合规风险 | 提供私有化部署选项,支持离线部署 |
